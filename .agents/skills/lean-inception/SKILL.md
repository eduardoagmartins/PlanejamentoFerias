# Lean Inception

Use this skill to help teams align around the right product, define an MVP, and turn discovery evidence into inputs that can drive Spec Kit or another spec-driven development flow.

## Core posture

- Treat Lean Inception as an alignment workshop for defining an MVP, not as generic discovery, portfolio prioritization, sprint planning, technical design, or process design.
- Treat the workshop as the discovery layer before Spec Kit: capture intent, evidence, assumptions, risks, and MVP boundaries before writing implementation-facing specifications.
- Keep the facilitation pragmatic: start from the user's context, adapt the agenda, and produce artifacts the team can act on immediately.
- Separate divergent moments from convergent decisions. Create space for shared understanding before narrowing scope.
- Make assumptions visible when the user gives limited context. Prefer a usable draft with explicit assumptions over blocking on perfect inputs.
- Preserve the language of the team. If the user writes in Portuguese, deliver workshop material in Portuguese.
- Do not promote every brainstormed feature into a requirement. Promote only features tied to personas, journeys, MVP proposal, expected outcomes, and validation metrics.

## Before acting

Ask only for missing information that changes the shape of the workshop materially. Useful inputs are product/problem, participants, workshop format, available time, current stage of discovery, constraints, desired output, and whether the next step is a Spec Kit `/specify`, `/plan`, `/tasks`, or implementation handoff.

If the user wants a complete plan and context is thin, assume:

- A one-week workshop with half-day sessions.
- Active participants from product, design, engineering, business, and user/customer knowledge.
- Stakeholders joining kick-off and showcase.
- The desired outcome is an MVP proposal with features, personas, journeys, success metrics, risks, assumptions, and a Spec Kit-ready discovery handoff.

## Decide fit

Recommend Lean Inception when the team needs to align a group around an MVP after enough discovery, ideation, experimentation, or strategic direction exists.

Recommend another approach when the user's need is primarily:

- Discovering user problems or doing research.
- Choosing among portfolio initiatives.
- Designing a prototype in a short sprint.
- Planning sprint-level user stories.
- Mapping detailed user story journeys.
- Defining technical architecture or engineering design.
- Aligning many teams on a program plan.
- Defining the team's way of working.

When Lean Inception is not the best fit, name the better method and explain briefly.

## Working flow

1. Frame the objective, decision to be made, expected MVP outcome, and non-goals.
2. Identify participant groups: active product team, invited stakeholders, and facilitator.
3. Choose agenda shape: full one-week, compressed, remote, or follow-up synthesis.
4. Run or prepare the core activities:
   - Kick-off
   - Product vision
   - Is / Is not / Does / Does not
   - Personas
   - User journeys
   - Feature brainstorming
   - Technical, UX, and business review
   - Features in journeys
   - Sequencer
   - MVP Canvas
   - Showcase
5. When the user mentions Spec Kit or Spec Driven Development, translate workshop outputs into a discovery handoff before producing or updating specs.
6. Produce the artifact requested by the user: agenda, facilitation guide, Miro/FigJam board outline, participant invite, activity prompts, synthesis report, MVP Canvas, roadmap sequence, Spec Kit discovery brief, or decision memo.
7. End with concrete next actions: validation experiments, `/specify`, `/plan`, `/tasks`, backlog/PBB, delivery planning, metrics setup, stakeholder decisions, and risks.

## Output standards

- Use workshop-ready wording, not abstract method explanation.
- Include timeboxes, objectives, inputs, activities, and outputs when creating agendas or facilitation guides.
- Make decision criteria explicit: user value, business value, learning value, feasibility, risk, dependencies, and effort.
- When synthesizing, separate facts, assumptions, open questions, decisions, and recommended next steps.
- Avoid treating the MVP as a feature dump. Emphasize the minimum scope needed to validate whether the initiative should continue.
- For Spec Kit, keep business/user language in the discovery and specify phases; move technical design choices to planning unless they are non-negotiable constraints discovered in the workshop.
- Mark every requirement seed with its source: vision, objective, persona, journey, feature card, sequencer, MVP Canvas, stakeholder decision, or assumption.
